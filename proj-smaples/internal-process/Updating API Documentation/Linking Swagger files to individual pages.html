Linking Swagger files to individual pages

Now that the files are uploaded to the dev-doc repo you'll need each corresponding documentation page to point back to those files, so they will display correctly when the live page loads.

To point each page to the correct swagger file

  1. First open one of the pages that points back to a corresponding Swagger file (users, tasks, projects, other, issues, general, documents). Notice that index.html is a landing/navigation page that doesn't display any Swagger information.
  1. Once you have a page file open, search the document for the <rapi> tag. Here is an example from the users.html file:
    <rapi-doc 
        spec-url="https://cdn.devdocs-dev.workfront-dev.com/res/swagger/swagger030320/attask-USER030320.json" 
        show-header = 'true'
        show-info = 'false'
        allow-authentication ='false'
        allow-server-selection = 'false'
        allow-api-list-style-selection ='false'
        theme = 'light' 
        layout = 'row'
        >
  1. Notice that under the spec-url parameter, the URL corresponds with the HTTPS location of the swagger files that have been uploaded to your AWS bucket:
    spec-url="https://cdn.devdocs-dev.workfront-dev.com/res/swagger/swagger030320/attask-USER030320.json"

    This format is used because the current version of Rapidoc that is being used on this site is a CDN and the external web-link needs to be used in order to avoid CORS conflicts.

  1. The prefix of the URL will remain consistent between all the pages:
    https://cdn.devdocs-dev.workfront-dev.com/res/swagger/
  1. Replace the last directories with the location of the corresponding swagger doc with the updated information. Notice that everything should remain the same except for the dates on the last folder level and the corresponding file should be updated to the most recent date:
    spec-url="https://cdn.devdocs-dev.workfront-dev.com/res/swagger/swaggerMMDDYY/attask-USERMMDDYY.json"
  1. Repeat this process for each page that has a Rapidoc tag.

After you've completed this process for all the display pages in the repo, upload your changes to the feature branch of the repo. To push your changes live to the AWS bucket, you can merge your changes with the master branch and then run the corresponding CI/CD pipelines. Notice that there is sometimes a significant lag between the times that code is pushed to the AWS bucket and when it will display live. Also notice that some code will update before other code on the site. Due to these inconveniences, it's best to wait 24 hours before assuming something has been broken.
