Pull Requests to the Master Branch

After your local changes have been pushed to the "stage" branch of the Redrock Swagger Gitlab repository, they will reside there. Other people working on the same project branch can pull the changes you've made, to their own local instances, and push their changes to the same branch, where you can, in turn, make a pull request to update your local instance.

These processes take place completely independent of the Master Branch, where a solid working copy of your repository should always exist. If someone pushes faulty data to the "stage" branch and the integrity of the API files are compromised, then the faulty data can be located and fixed on the "stage" branch, without corrupting the integrity of the working master branch.

When a significant amount (a quantity determined by all people working on the repo) of changes have been made to the "stage" branch—and the branch is stable—then a pull request can be made from the master branch, to the "stage" branch, so that your master files can be made current with the working project branch.

Making a pull request from the master branch

There are several different ways to do this, but for simplicity, the writer with the highest privileges on the Redrock Swagger Repository should make the request by accessing Gitlab through a web browser.

  1. From the main repository overview, select the "merge request" icon from the menu at the left-hand side of the screen.
  1. The next page will prompt you to specify the "source" branch and the "target" branch.
    1. If you are updating the master branch, then the "stage" branch is your source and the master branch is your "target".
    1. If you are attempting to replace information on the "stage" branch with information from the master branch—and you are not intimately familiar with Gitlab merge requests—then seek the assistance of a friendly developer or engineer. They should be able to help you quickly resolve your situation.
  1. Don't worry about the other options at this time. Once you have specified the source and target, submit the information and move on to the next page.
  1. The next page will prompt you to leave a description for your merge request. Generally speaking, this description will most often be something to the effect of: "Updating descriptions for the endpoints of the (object name) object." Or any other task that you have performed since the last merge.
  1. The only other option that is necessary to specify should be the requester's name, which should be set to you, or whoever is making the merge request.
  1. Once you have submitted the data from this page, your merge request will be created. Note: At the time that this documentation was created there was only one writer using and maintaining this repository. So, by default, the merge request doesn't require any sign off from another repo user. However, if at any time there is more than one writer contributing to this repo, merge requests should be set to have at least one approval from another team member. This way, more eyes can ensure the integrity of the stage branch before the data is pushed to the master.
  1. If no approvals are required to make the merge request go through, you should be able to press the green merge button to initiate the merge. When merging with the master branch, you should expect there to be no merge conflicts. It isn't impossible that you might experience some, but it is unlikely.
  1. If you suspect that something is wrong after the merge went through, Gitlab should offer you the option to revert the branches back to their original state. If you revert the branches:
    1. locate and fix the problems on the "stage" branch.
    1. perform a new merge request and document the changes that were made to correct any errors in the merge request description.
